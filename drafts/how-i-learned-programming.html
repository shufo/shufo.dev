私がどのようにプログラミングを学んだか

Shufo | 2019 Jan/06 (Sun)

プログラミングをどのように学ぶかという記事が話題になっていたのでちょうどいい機会なので自分がプログラミングに触れてから今までのことを振り返ってみる

PC との出会い, HTML, CSS

最初に自分用の PC を手に入れたのは割と遅く、高校の入学祝いだった。 それまで PC を持てなかった反動でのめり込むように PC にハマったのを覚えている。

確かちょうど 00 年台の中頃で、大文字の WEB がまだ微かに未来に希望を抱いていた頃だ。梅田望夫の ウェブ 進化論などが話題になっていた。

古きよきインターネットの例に漏れず自分用の HP とかブログを作っていた。 その過程で HTML や CSS に触れたのが初めてのプログラミングらしいプログラミングといえるかもしれない。(HTML5+CSS はチューリング完全という説もあるので…) そのうちブログを書くことより HTML や CSS をいじってカスタマイズする方にハマりブログの方は殆ど更新されなくなった。

専門学校入学, IPA 試験

その後なんやかんやあり（雑）高校卒業後、ニート期間を経て専門学校に入学。 この頃のニート期間は暗黒なのであまり思い出したくない(暗黒微笑)

専門学校は半ばモラトリアムのためといえなくもないけど当時はやりたいことも特に無かったので IT 系の学科に入学した。

初めてちゃんとしたプログラミングに触れたのはこの時で最初にやったのは C 言語 だった。 コンソールで文字列を出力したりソートしたりといったような基本的な内容だったと思う。正直おもしろくはなかったのであまり内容は覚えていない。

今思うとその専門学校の 1 期生として入学したこともありカリキュラムも手探りみたいな感じで、正直あまり体系的な授業とはいえなかったのが少し残念だった。老舗の専門学校とかならもう少し体系的なカリキュラムも整っていたかもしれない。 ただプログラミング自体はあまり実用的な知識は身につかなかったけど、プロジェクト形式でチームを組んで進めたり DB の設計などのレビューを受けたりといった授業もあったのでそういったところは参考になった。

そういった中で個人的に専門学校で最も役に立ったのは資格試験の勉強だった。

いわゆる IPA の基本情報などの試験のことだけど、年二回開かれる IPA の試験 1 週間ぐらい前から試験期間となり通常授業ではなく試験対策用の授業になっていた。（といっても授業内容が試験範囲の講義に変わるくらい）

IPA の資格は Web 業界では正直評価されることは少ないが、個人的には素人が基礎力を付ける最も体系的な方法だと思う。 というのは基本情報や応用情報などで扱われる話というのはプログラミングの基本概念やアルゴリズム（ソートとか）であって特定の言語によらない知識を身につけることが出来るからだ。

そして試験に合格するためにはもちろん試験勉強が必要で、この過程で何度となく反復をすることで基礎が身につく。

正直最短の道とは言えないし今すぐお金になる知識でもないけど長い目で見ると今も結構助かっていることが多い。結局在学中は基本情報、応用情報、ネットワーク、データベース、情報セキュリティの高度 3 種を取ったけどその時の知識が今でも血肉になって新しい技術を学ぶ時の土台になっている。

具体的には

  • 基本情報: IT 全体の本当に基本的な知識(ビット, バイト, CPU, メモリ, アルゴリズムなど)
  • 応用情報: 基本情報+α の知識(プロジェクトマネジメント, 開発手法等, トランザクション, ネットワークの基本理論等)
  • データベーススペシャリスト: DB のモデリングや設計
  • ネットワークスペシャリスト: ネットワークの設計、TCP/IP スタックに関する知識等
  • 情報セキュリティスペシャリスト: セキュリティのリテラシー

といったところを学べるがこのあたりを広く浅く学ぶことで全体的なシステム観が身についた。

個人的にこういった広く浅く学ぶことが重要なのはプログラミングというものも問題解決の一手段に過ぎないということが学べることだと思う。

確かに圧倒的なハッカー的プログラミングで問題をズバっと解決するというのは夢があるけど、現実のシステムで相手をするのは複合的な要素で、得てしてプログラミングを一切せずに問題を解決するということが可能なことも多いのだ。

例えば要件から得られる DB のモデリングや設計さえちゃんとしていれば大抵のアプリケーションはコーディングがこじれることもなく綺麗に設計出来るし、プログラミングで解決しなくてもネットワークの設計を見直すことで解決可能な問題も多い。プロジェクトマネジメント手法や話し合い一つで解決出来る問題すらある。またセキュリティの知識無しにシステムを組むのは無謀だ。

そういった大局観を持つためには応用的な知識ではなく基礎的な知識の組み合わせが必要でプログラミングで無理やり解決しようとすると問題がこじれる場合すらある。

そのために基本情報や応用情報などの試験は本当にずぶの素人がこれから IT 業界をサバイブしていくというような場合に有用だと思う。 IT 業界ではまず知ってるか知らないかだけというような知識も多くあり、知らないがために見当違いなソリューションで無駄に苦労するという事例も枚挙に暇がない。

ちなみに基本情報や応用情報は知っておいて当たり前な知識の網羅なので持っているからといって Web 業界などではすぐ評価されるというものでもないが、 仮にも国家資格なので国外で働くことを想定している場合にはその国の同等の試験に受かっているものとして試験の相互認証を受けれたりといった優遇措置はあったりする。

社会人、初めてのまともなプログラミング

そして卒業後、入社した会社で初めてまともに複数人でプログラミングする現場を見てプログラミングというものをちゃんと考えるようになった。 在学中にまともにプログラミングといえるプログラミングをしたのは PHP や CakePHP などのフレームワークぐらいで、その時は共同作業も少なかったし正直動けばいいぐらいに思っていた。

そんな感じで入社した初めての会社でも Web 業界の例に漏れず教育らしいものはなかったので今思えば最初の頃はそれはひどいコードを書いていて、初めて配属された 2 プロジェクトくらいはそれこそ動いてるから OK という意識でさえあったと思う。

その後初めてちゃんと設計を意識したのは名も知らぬ誰かが書いたコードで確か他社からコードを引き継いだ保守案件だった。 コミットログも残っていないままだったので今でも誰が書いたのかは分からないが PHP のプロジェクトで案件の内容に見合わないほど整然としたコードだった。

他のコードと違ったのはコードを読んだだけで設計者の意図も分かってどこを変更すれば保守出来るかが手に取るように分かった点だ。 そういったコードを実際の複雑性を持つ実プロジェクトのコードで見たのは初めてだった。 そのコードに衝撃を受けてやっと設計の重要さに目覚めた。 個人的に今でも心のプログラミングの師匠と勝手に呼んでいる。

息を吸うようにプログラミングをする根っからのプログラマーという訳でもなく、あくまで問題解決技法としてのプログラミングを楽しんでいた者としてある意味目覚めの経験になったのはこんな感じで大分遅く、働き始めてからだったというわけだ。

結局プログラミングに関しては誰かに教えられたという訳ではなく、名も知らぬ誰かのコードからプログラミングを学ぶ self-tought なエンジニアの一員なので未だにプログラミングをどう教えるかという体系だった方法は分からないが学生時代の同級生もプログラミングが得意な人は self-tought な人が多かったと思う。

しかしこれをやってはっきりよかったと言えるのはパブリックな領域にコードを公開することとテストを書くこと、人のコードを読むことだった。

プログラミング上達に近道はないけどこの 2 つだけが自分のプログラミングに良い影響を与えた習慣だと強く思う。
