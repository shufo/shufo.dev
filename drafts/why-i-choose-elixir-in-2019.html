人間工学としての Elixir

Shufo | 2019 Jan/07 (Mon)

つい先日 HN で Elixir を選ぶ理由についての記事が人気が集まっていた

Would you still pick Elixir in 2019? | Hacker News

Elixir を選ぶ理由と他言語との比較が GitHub issue に上げられていてそれについてのコメントが集まっている

自分が気になったのはこのコメント欄で何度か出て来る

Programmer Ergonomics

という単語で、この単語で自分がぼんやりと抱いていた、 Elixir を一言で説明するための方法が見つかった気がする

ぼんやり抱いていたというのは今までどうもうまく自分の中で Elixir が好きな理由を言語化出来ていなかったのだ

Elixir を説明する単語として productivity, scalable といった単語がよくセールスポイントになるし、実際それを説得材料としても使ったことはあるがそれはあくまで売りだすための PR ポイントで本質的に何にフォーカスしているかということに答えていない気がしていた

けどやっとこの Ergonomics という単語を見てフォーカスしているのは人間ではないか？ということに気づいた

人間とプログラミング言語のインターフェースはもちろん言語の文法であり、ランタイムであり、コンパイラであり、またエコシステムやエディタ、コミュニティでもある

それら諸々合わせた総合的なインターフェースが人間にフォーカスして作られている

つまり Elixir を一言で説明すると ergonomic language ではないかということだ

ergonomics の定義は国際人間工学連合(IEA）によると

人間工学とは、システムにおける人間と他の要素とのインタラクションを理解するための科学的学問であり、人間の安寧とシステムの総合的性能との最適化を図るため、理論・原則・データ・設計方法を有効活用する独立した専門領域である

とある

つまり人間の認知と構造を理解し、人間とコンピュータのインターフェースにおける不協和を解消するための言語ということだ

それだけ言ってもなんのこっちゃなので以降でそう思った理由を説明する。

自分は Elixir をやるまでは正直プログラミングを楽しいと思ったことはなかった

それまでの経験としては C と C#を軽くと Python, JavaScript, PHP などをやってきたがどれも「楽しい」という感覚を得られるまでにはならなかった。それらはあくまで仕事の道具でありいわばトンカチのようなもので仕事の相棒ではあるけど、「楽しいから」プログラミングをするという状態ではなかった

それらのどの言語も「自分の言語」ではないという感覚を抱いており、なんとなく違和感を感じていた

その中で一番好きな Python でさえよく出来たトンカチのようなもので叩けさえすれば形にはこだわらない、というような気持ちだった

言語を「使う」というよりは「使われている」という感覚に近く、確かにプログラミングしているのは自分だけどその言語の文法に従っているだけで決まりきった一本道を歩いているだけのようなそんな感覚だった。

それが Elixir をやって初めてプログラミングをやって「楽しい」という感覚に変わった

ただ書かされるだけじゃなく自分から書くのが楽しいと思える言語は初めてだった。

そう思えたのはおそらく自分の中でこの言語は「自分の言語」だという感覚を得られたからだと思う。例えば日本語話者が英語を書くときに一旦頭の中で翻訳して書くように、自分の自然な感覚から疎外された感覚を覚えていた。

それは言語の積み重ねたバックグラウンドや歴史もあるだろうけど、しっくりくるという感覚ではなかった。

自然言語と異なり形式言語のプログラミング言語には必ず設計者の意図が介在する。その意図は政治的要因で決められることも少なくない。そして政治的要因で決められた仕様は必ず禍根を残す。

歴史が深い言語ほどこの問題は大きく、ハイコンテクストな文化になっていく。

新しい言語は「比較的」政治的要因が少ないからマシなだけだ。もちろん新しくても政治的な言語はあるし、古くても慎重に政治性を排除した言語はある。

政治的要因が絡むと純粋な形式言語世界から飛び出して「私の」言語ではなくその他の利害関係含む「私たち」の言語になる。そういった状況が嫌だったんだと思う。

もちろん政治的動物である人間である以上プログラミング言語の政治性からも免れ得ないが、形式言語の世界だけは出来るだけ政治性を排したい。

そういった点で Elixir は慎重に選択をしている言語だ。

作者は思慮深くコミュニティに耳を傾けて特定の政治的配慮に与しない。

プログラミング言語は歴史

科学的な論証は何もないけどなぜ Elixir はしっくりくるという感覚を得れたのか。

以下に言語の複雑性についての記事があった。

How to Measure Programming Language Complexity – Richard Kenneth Eng – Medium

この記事では言語の予約語やキーワードの数を数えることで言語の複雑性を表すという方法を取っている。

これによると Elixir はたったの 18 個で Smalltalk に次いで二番目に予約語の少ない言語だった。

それもそのはず、Elixir では if のような一見制御構造に見えるキーワードもただのマクロだったりするのだ。これによって言語のコア自体はとてもシンプルだけどとても自由度の高いことが出来るようになっている。例えば if を override して true だったら else 側を実行するなんてことも出来る

教育用言語として考えると Smalltalk の予約語の少なさはある意味理想的かもしれないが、 Real-World の使用の多さでいえばある意味最も予約語の少ない（覚えることのすくない）汎用言語は Elixir と言えるのではないか

限りなくシンプルにした Go 言語でさえ 25 個なのだからそれより少ないというのは正直驚きだ

これにより言語がとても習得しやすくなっている

実際に私が会社に Elixir を導入し実案件で運用した際、後から入ってきた Elixir 未経験の人が 2 週間程ですぐ立派な Elixir コードを書いていたのをよく覚えている
